# Multimodal Light Agent

一个基于 Python、OpenCV 与 LLM 接口的轻量级多模态 Agent 原型系统。

本项目面向视频、图像和文本等基础多模态输入，尝试构建一个从自然语言问题输入、模态识别、任务路由、工具调用到最终回答生成的轻量级 Agent 执行流程。当前版本以规则路由和工具调用为核心，视频模态能力相对完整，同时预留 LLM 问题理解与工具选择接口，并支持通过大模型对工具执行结果进行自然语言重写。

> 项目持续开发中，当前版本主要用于验证多模态工具调用、任务路由和轻量级 Agent 执行链路设计。

---

## 1. 项目定位

本项目不是完整的视频理解大模型系统，而是一个轻量级多模态 Agent 原型。

当前系统重点关注以下问题：

- 如何根据用户自然语言问题识别任务所属模态；
- 如何将不同模态下的处理能力封装为可注册、可调用的工具；
- 如何通过规则路由完成任务分发；
- 如何在工具执行后生成统一格式的自然语言回答；
- 如何为后续 LLM 工具选择、意图理解和多模态扩展预留接口。

当前版本的整体流程如下：

```text
用户问题输入
    ↓
LLM 问题理解入口
    ↓
模态识别
    ↓
子任务路由
    ↓
工具注册表查询
    ↓
工具执行
    ↓
LLM 回答重写
    ↓
最终回答输出
```

## 2. 当前功能

### 2.1 视频模态

视频模态是当前版本的主要功能模块，支持以下能力：

| 功能 | 说明 |
|---|---|
| 视频时长统计 | 读取视频总帧数和 FPS，计算视频总时长 |
| 视频帧数统计 | 获取视频总帧数 |
| 视频帧率统计 | 获取视频 FPS |
| 运动变化检测 | 基于帧差方法判断视频中是否存在明显画面变化 |
| 明显变化时间定位 | 基于帧差峰值定位视频中变化较明显的时间点 |
| 视频内容初步分析 | 返回规则化的视频内容分析结果 |
| 视频摘要 | 返回规则化的视频概括结果 |

当前视频分析部分主要基于 OpenCV 实现，采用轻量级规则方法，不依赖复杂深度学习模型。

---

### 2.2 图像模态

当前图像模态支持基础图像属性读取：

| 功能 | 说明 |
|---|---|
| 图像尺寸读取 | 获取图片宽度和高度 |

---

### 2.3 文本模态

当前文本模态支持基础文本统计：

| 功能 | 说明 |
|---|---|
| 字符长度统计 | 统计文本字符数量 |
| 词数统计 | 基于空白切分规则统计文本词数 |
| 句子数统计 | 基于常见中英文句末标点粗略统计句子数量 |

## 4. 核心设计

### 4.1 规则路由

系统首先根据用户问题判断任务所属模态：

- 包含“视频”或 `video` 时，识别为视频任务；
- 包含“图片”“图像”“照片”等关键词时，识别为图像任务；
- 其他情况默认作为文本任务处理。

在识别出模态后，系统会继续根据关键词判断具体子任务。例如：

| 用户问题关键词 | 路由任务 |
|---|---|
| 时长、多久、duration | 视频时长统计 |
| 帧数、总帧数 | 视频帧数统计 |
| fps、帧率 | 视频帧率统计 |
| 运动、活动 | 视频运动变化检测 |
| 什么时候变化、变化最明显 | 明显变化时间定位 |
| 尺寸、宽高 | 图像尺寸读取 |
| 长度、多少字 | 文本长度统计 |
| 词数、word count | 文本词数统计 |
| 句子数、sentence count | 文本句子数统计 |

---

### 4.2 工具注册表

项目将不同模态下的处理能力统一封装为工具，并在 `tools.py` 中进行注册。

每个工具包含以下元信息：

```text
name
description
modality
input_type
output_type
keywords
example_question
func
```

这种设计使系统具备较好的可扩展性。后续如果需要增加新工具，只需要在对应模态的工具库中注册新的工具信息，并绑定具体执行函数即可。

---

### 4.3 LLM 接口层

项目中设置了统一的 LLM 接口层，当前主要用于对工具执行结果进行自然语言重写。

当前支持的 provider 设计包括：

| Provider | 状态 |
|---|---|
| mock | 默认模式，不调用外部模型，直接返回原始结果 |
| openai | 已预留接口 |
| deepseek | 支持通过 DeepSeek API 对工具结果进行回答重写 |

如果未配置真实 API Key，系统会自动回退到原始工具结果，从而保证基础流程仍可运行。

需要注意的是，当前版本的 LLM 工具选择接口已经预留，但尚未作为主要决策逻辑启用。实际工具选择仍主要依赖规则路由完成。

## 5. 安装与运行

### 5.1 环境要求

建议使用 Python 3.8 及以上版本。

### 5.2 安装依赖

```bash
pip install -r requirements.txt
```

```text
opencv-python

python-dotenv
```

---

## 6. 环境变量配置

如果只使用默认 mock 模式，不需要配置 API Key。

如果需要启用 DeepSeek 回答重写能力，可以在项目根目录创建 `.env` 文件：

```env
LLM_PROVIDER=******(deepseek)
DEEPSEEK_API_KEY=******(your_api_key_here)
DEEPSEEK_MODEL=******(your_model_name_here)
```

如果不配置 `LLM_PROVIDER`，系统默认使用 `mock` 模式。

---

## 7. 使用方式

运行主程序：

```bash
python main.py
```

程序会先要求输入自然语言问题，例如：

```text
请输入你的问题：这个视频有多长？
```

如果系统识别为视频任务，会继续要求输入视频路径：

```text
请输入视频文件路径：C:\Users\xxx\Desktop\test.mp4
```

系统随后会输出识别结果、可用工具、执行决策、工具执行结果和最终回答。

## 9. 当前边界

当前项目仍处于轻量级原型阶段，主要存在以下边界：

1. 视频内容理解主要基于 OpenCV 和规则方法，暂未接入真正的视频语义理解模型；
2. 工具选择主要依赖关键词规则，LLM 工具选择接口已预留但尚未作为主流程启用；
3. 图像模态当前只支持尺寸读取，尚未扩展目标检测、图像描述等能力；
4. 文本词数统计采用空白切分规则，对中文连续文本只是轻量近似；
5. 视频摘要和视频内容分析目前仍是规则化输出，后续可以进一步结合视觉模型或多模态大模型增强。

## 10. 后续规划

后续可以从以下方向继续扩展：

- 接入真实 LLM 工具选择逻辑，使模型根据 tool schema 自动选择工具；
- 增加工具选择结果校验，避免模型输出不存在的工具名；
- 扩展图像模态能力，例如图像分类、目标检测、图像描述等；
- 扩展视频模态能力，例如关键帧提取、目标运动轨迹分析、动作识别等；
- 增加统一日志模块，记录问题、模态、工具选择、执行结果和最终回答；
- 增加命令行参数或简单 Web UI，提高项目展示效果；
- 将规则路由、工具调用和 LLM 推理进一步解耦，形成更完整的轻量级 Agent 框架。
